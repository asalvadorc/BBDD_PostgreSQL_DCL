🧪 PRÁCTICA: Gestión de Transacciones con TCL

🎯 Objetivo

Aprender a:

  • Utilizar BEGIN / START TRANSACTION
  • Aplicar COMMIT
  • Aplicar ROLLBACK
  • Usar SAVEPOINT
  • Entender la importancia de la atomicidad en bases de datos

🗂️ Contexto

Trabajaremos con una base de datos de ventas de entradas de cine.

Cuando un cliente compra una entrada, deben realizarse varias acciones:

  1. Registrar la compra
  2. Marcar el asiento como ocupado
  3. Reducir el número de asientos disponibles

⚠️ Todo esto debe hacerse como una única unidad lógica.

🏗️ PREPARACIÓN

Crea un esquema propio llamado: app

📦 Crear tablas

Dentro del esquema app, crea:

Tabla Sesiones

  • id_sesion: INT
  • pelicula: VARCHAR(50)
  • asientos_disponibles: INT

Tabla Asientos

  • id_asiento: INT
  • id_sesion INT
  • ocupado: BOOLEAN

Tabla Ventas

  • id_venta: INT
  • id_sesion INT
  • id_asiento: INT

📊 Insertar datos iniciales

  • Sesiones: Introduce la primera sesión con 5 asientos disponibles en la película 'Avatar 3'.
  • Asientos: Introduce 5 asientos de la sesión 1.

🧩 PARTE 1 — Sin Transacción (Para ver el problema)

  • Simular una compra SIN transacción:

    • Inserta una Venta del asiento 2 de la sesión 1.
    • Actualiza el asiento vendido como ocupado
    • Simula un fallo
    • Actualiza los asientos disponibles de la venta.

  • Consultar el estado en el que han quedado las tablas.

📌 Pregunta:
¿Es consistente la base de datos? Razona la respuesta.

🧩 PARTE 2 — Uso de BEGIN + COMMIT

  • Repite las operaciones anteriores, cambia algún dato si fuera necesario para no infringir la integridad referencial, pero ahora introdue COMMIT.

  • Consultar el estado en el que han quedado las tablas.

🧩 PARTE 3 — Uso de ROLLBACK

  • Repite las operaciones anteriores, cambia algún dato si fuera necesario para no infringir la integridad referencial, pero ahora introdue ROLLBACK.

  • Consultar el estado en el que han quedado las tablas.

📌 Pregunta:
¿Se ha realizado alguna modificación?

🧩 PARTE 4 — Uso de SAVEPOINT

  • Repite las operaciones anteriores, cambia algún dato si fuera necesario para no infringir la integridad referencial, pero ahora introdue un SAVEPOINT antes de cada actualización.

  • Consultar el estado en el que han quedado las tablas.

📌 Pregunta:
¿Qué cambios se han guardado?

🧠 PARTE 5 — Reflexión

Responder:

  1. ¿Qué ventaja tienen las transacciones?
  2. ¿Qué ocurriría en un sistema real sin ellas?
  3. ¿Qué diferencia hay entre ROLLBACK y ROLLBACK TO SAVEPOINT?
  4. ¿En qué situaciones reales usarías SAVEPOINT?

🏁 RETO FINAL

Crear una transacción que:

  • Venda 2 entradas a la vez
  • Si falla una, no se venda ninguna
